# credit-risk-scorer

Credit risk scoring API trained on the German Credit dataset. Predicts loan default probability using XGBoost, tracked with MLflow, served via FastAPI. Every prediction includes a plain-English explanation of the factors driving the risk score. Deployed on Azure App Service.

## 1. Quick Start

### Path A вЂ” Local with Docker Compose (recommended)

```bash
git clone https://github.com/xavier-oc-programming/credit-risk-scorer
cd credit-risk-scorer

# Train the model first (requires Python env with requirements installed)
pip install -r requirements.txt
python train.py

# Start API + MLflow server
docker-compose up
```

- API: http://localhost:8000/docs
- Demo UI: http://localhost:8000/demo
- MLflow: http://localhost:5000

### Path B вЂ” Local without Docker

```bash
pip install -r requirements.txt
python train.py
uvicorn main:app --reload
```

API available at http://localhost:8000/docs

### Path C вЂ” API only (models/ already present)

If you have trained model files in `models/`:

```bash
pip install -r requirements.txt
uvicorn main:app --reload
```

> **Note:** `models/` is gitignored вЂ” trained model files are not committed to the repo. They are generated by `python train.py`. The CI pipeline tests endpoint structure and validation using FastAPI's TestClient, which does not require real model files for health/docs/validation tests. For `/score` endpoint tests, models must be present locally.

> **Dataset note:** the training script fetches the German Credit dataset from OpenML. OpenML is a free academic service and is occasionally slow or unresponsive. The code automatically falls back to downloading directly from the UCI ML Repository if the OpenML ping times out after 5 seconds вЂ” the dataset is identical either way. After the first successful download the data is cached locally in `~/scikit_learn_data/` and no network request is made on subsequent runs.

## 3. Dataset

**German Credit (Statlog)** вЂ” 1,000 loan applicants, 20 features.

- Source: UCI Machine Learning Repository, loaded via `sklearn.datasets.fetch_openml('credit-g')`
- Target: `class` вЂ” `good` (700 applicants) or `bad` (300 applicants)
- Class imbalance: 70/30 split

The imbalance matters for evaluation. A model that approves every application achieves 70% accuracy while correctly identifying zero bad loans. Accuracy is therefore a misleading metric for this problem вЂ” the models are evaluated on cost-weighted score and recall on the bad class.

Key predictive features: `checking_status`, `duration`, `credit_history`, `credit_amount`, `savings_status`, `employment`, `age`, `purpose`, `installment_commitment`.

---

## 4. Models

Three classifiers are trained and compared:

| Model | Description |
|---|---|
| **Logistic Regression** | Interpretable linear baseline. Fast to train. Assumes linear decision boundary вЂ” unlikely to capture all interactions in credit data. |
| **Random Forest** | Ensemble of decision trees. Handles non-linear relationships and feature interactions. Less sensitive to scaling than LR. |
| **XGBoost** | Gradient boosting. Typically strongest on structured/tabular data. Uses `scale_pos_weight=5` for class imbalance instead of `class_weight`. |

**Model selection metric:** `cost_weighted_score` вЂ” not ROC-AUC. See Section 12 for the rationale.

## 5. Results

| Model | Accuracy | Precision | Recall | F1 | ROC-AUC | Cost-Weighted |
|---|---|---|---|---|---|---|
| **Logistic Regression** вњ“ | 0.615 | 0.430 | **0.867** | 0.575 | **0.803** | **0.752** |
| Random Forest | 0.760 | 0.688 | 0.367 | 0.478 | 0.789 | 0.546 |
| XGBoost | 0.715 | 0.525 | 0.533 | 0.529 | 0.760 | 0.616 |

**Winner: Logistic Regression** вЂ” selected by `cost_weighted_score`. Despite lower accuracy than Random Forest, it catches 86.7% of bad loans (recall) vs 36.7% for Random Forest. In credit lending, an undetected bad loan costs 5Г— more than a rejected good loan вЂ” high recall on the bad class is the objective.

## 6. SHAP Explainability

Every `/score` response includes plain-English risk factors generated by `risk_explainer.py`. The module translates raw SHAP values into statements a loan officer would recognise.

Example output for a high-risk applicant:

| Feature | SHAP value | Plain-English risk factor |
|---|---|---|
| `checking_status_no checking` | +0.38 | no checking account increases default risk |
| `duration` | +0.29 | longer loan duration increases default risk |
| `credit_history_delayed previously` | +0.24 | previous payment delays significantly increase default risk |
| `savings_status_>=1000` | в€’0.18 | high savings significantly reduce default risk |
| `employment_>=7` | в€’0.11 | long-term employment significantly reduces default risk |

`risk_explainer.py` separates features into:
- **Risk factors** вЂ” SHAP > 0, sorted by magnitude (top 3 returned)
- **Protective factors** вЂ” SHAP < 0, sorted by magnitude (top 3 returned)

Features with `|SHAP| < 0.01` are omitted.

## 8. API Reference

The primary documentation is the auto-generated OpenAPI UI at **/docs**.

### `POST /score`

Score a single credit application.

**Request:**
```json
{
  "duration": 24,
  "credit_amount": 5000,
  "installment_commitment": 3,
  "age": 35,
  "existing_credits": 1,
  "checking_status": "0<=X<200",
  "credit_history": "existing paid",
  "purpose": "new car",
  "savings_status": "<100",
  "employment": "1<=X<4"
}
```

**Response:**
```json
{
  "probability_of_default": 0.4823,
  "risk_score": 48,
  "risk_band": "High",
  "decision": "Additional checks",
  "risk_factors": [
    "low checking account balance increases default risk",
    "longer loan duration increases default risk",
    "new car purchase increases default risk"
  ],
  "protective_factors": [
    "existing credits paid on time reduces default risk"
  ],
  "model_used": "XGBoost",
  "shap_values_raw": { "duration": 0.1823, "credit_amount": 0.0912, "..." : "..." }
}
```

### `POST /score/batch`

Score up to 100 applications in a single request. Accepts a JSON array, returns a JSON array of the same length.

### `GET /health`

```json
{ "status": "ok", "model_loaded": true, "model_name": "XGBoost" }
```

### `GET /api/model-info`

Returns all three models' metrics with `"primary_metric": "cost_weighted_score"`.

### `GET /api/feature-importance`

Returns the top 10 SHAP features with mean absolute importance values.

### `GET /api/risk-bands`

Returns the scoring thresholds: Low (0вЂ“20), Medium (20вЂ“40), High (40вЂ“60), Very High (60вЂ“100).

For interactive documentation, field descriptions, and example payloads: **/docs**

## 10. Deployment вЂ” Azure App Service

FastAPI is an ASGI framework and requires gunicorn with `UvicornWorker` on Azure App Service вЂ” different from the standard gunicorn command used for Flask.

```bash
# Create resource group and plan
az group create --name credit-risk-rg --location westeurope
az appservice plan create --name credit-risk-plan \
  --resource-group credit-risk-rg --sku B1 --is-linux

# (Scale to F1 via portal after creation)

# Create web app
az webapp create --name credit-risk-scorer \
  --resource-group credit-risk-rg \
  --plan credit-risk-plan \
  --runtime "PYTHON:3.11"

# Set FastAPI startup command
az webapp config set --name credit-risk-scorer \
  --resource-group credit-risk-rg \
  --startup-file "gunicorn main:app --workers 1 --worker-class uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000 --timeout 600"

# Enable build during deployment
az webapp config appsettings set --name credit-risk-scorer \
  --resource-group credit-risk-rg \
  --settings SCM_DO_BUILD_DURING_DEPLOYMENT=true

# Deploy
cd credit-risk-scorer
zip -r deploy.zip . -x "*.git*" -x "venv/*" -x "__pycache__/*" \
  -x "*.ipynb_checkpoints*" -x "mlruns/*"
az webapp deployment source config-zip \
  --name credit-risk-scorer \
  --resource-group credit-risk-rg \
  --src deploy.zip
```

## 12. Business Context

In credit lending, prediction errors have asymmetric costs:

- **False negative** (approving a bad loan): the bank risks losing the entire principal. Cost: high.
- **False positive** (rejecting a good loan): the bank loses a customer and interest revenue. Cost: lower.

The standard assumption in the German Credit dataset literature is that **a bad loan costs 5Г— more than a missed good loan**. This asymmetry is built into training in two ways:

1. `class_weight={0: 1, 1: 5}` passed to scikit-learn models вЂ” the loss function penalises false negatives more heavily during training.
2. `scale_pos_weight=5` for XGBoost вЂ” equivalent mechanism for gradient boosting.

And into evaluation via the cost-weighted score:

```
cost_weighted_score = 1 - (5 Г— FN + FP) / (5 Г— total_bad + total_good)
```

A model that minimises this score is directly optimising for the bank's actual cost, not a symmetric discrimination metric.

## 13. Design Decisions

**Why FastAPI over Flask for an API-first project.** This project's primary interface is `POST /score` called programmatically вЂ” a bank integrating this model does not use a form. FastAPI's auto-generated OpenAPI docs at `/docs` make the API immediately explorable and testable without a separate documentation step. Pydantic models validate every input field automatically and return structured 422 errors with field-level detail. Flask requires manual validation logic for the same behaviour. FastAPI is also ASGI-native, which enables `POST /score/batch` to handle concurrent requests вЂ” relevant for bulk loan decisioning workflows.

**Why `cost_weighted_score` over ROC-AUC for model selection.** ROC-AUC measures discrimination ability вЂ” how well the model ranks positive above negative cases вЂ” but does not directly measure business cost. The cost-weighted score penalises false negatives 5Г— more than false positives, reflecting the actual economics of credit lending. A model with lower ROC-AUC but higher cost-weighted score is preferable for deployment because it minimises the bank's expected loss, not a symmetric ranking metric.

**Why `class_weight={0:1, 1:5}` and `scale_pos_weight=5`.** The 70/30 class imbalance means an untrained model can achieve 70% accuracy by always predicting good. Reweighting shifts the model's attention toward the minority class (bad loans) during training, making false negatives more expensive in the loss function. The 5Г— weight reflects the assumed cost asymmetry rather than the class frequency ratio вЂ” it's a business parameter, not a statistical correction.

**Why Pipeline wraps preprocessor and classifier together.** The Pipeline ensures the StandardScaler is fit only on training data and applied consistently to test data and new predictions вЂ” preventing data leakage. It also means the saved `best_model.pkl` file includes the full preprocessing logic, so the API can accept raw input (a `CreditApplication` with string categorical values) and produce predictions without a separate preprocessing step.

**Why `risk_explainer.py` is a standalone module.** Explainability is not optional in credit scoring вЂ” the EU AI Act and US Equal Credit Opportunity Act both require that adverse credit decisions can be explained to the applicant. Isolating the SHAP-to-language translation in its own module makes it independently testable, replaceable, and auditable. The function signature (`shap_vals`, `feature_names`, `top_n`) is generic enough to work with any model's SHAP output, not just XGBoost.

**Why Docker Compose even for a single-model project.** Running the API and MLflow tracking server as separate containers mirrors a production microservices architecture where the scoring service and experiment tracking service are independent concerns. It also eliminates the need to run two processes in separate terminals during development. The volume mounts mean model artefacts trained on the host are immediately available to the containerised API without rebuilding.

**Why GitHub Actions CI tests endpoints, not training.** `train.py` downloads the German Credit dataset, trains three models, and runs SHAP analysis вЂ” taking several minutes and requiring network access. Testing it in CI adds cost and latency with low signal, since training failures are immediately visible locally. The CI pipeline tests what can regress silently: endpoint contracts, input validation behaviour, and import correctness. A production system would add model performance regression tests that compare a newly trained model against a performance floor before allowing deployment.
